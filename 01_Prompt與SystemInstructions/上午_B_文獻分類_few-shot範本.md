# 上午 B · 文獻分類：Few-shot 提示範本

> **核心觀念**：**Few-shot（少樣本示範）**＝「不用寫程式，只要先示範 2～3 個『輸入→輸出』的例子，AI 就會照樣把後面成千上百筆文字自動整理成結構化欄位。」
> **使用時機**：上午第一個實作主題，帶出「AI 幫你做量化編碼／內容分析的前處理」。
> **放哪裡**：整段 prompt 貼進 AI Studio 對話框（或把「角色與規則」放進 System Instructions、把範例＋待處理文字放對話框，兩種都可）。

---

## 一、建議參數

| 參數 | 建議值 | 理由 |
|---|---|---|
| **Temperature** | **0.1 ~ 0.3** | 分類 / 抽取是「照規則歸位」，要**一致、可重現**，不要發散。低溫讓同一筆文字每次分類結果穩定。 |
| Model | 較新 Gemini | 指令遵循佳；批次多筆時穩定。 |
| Max output tokens | 調高 | 批次輸出多筆時避免截斷。 |

---

## 二、要抽取的欄位（Schema）

| 欄位 | 說明 |
|---|---|
| **案由** | 這件救濟案在爭什麼（一句話） |
| **稅目** | 如：綜合所得稅、營業稅、營利事業所得稅、遺產稅、贈與稅、地價稅、房屋稅等 |
| **爭點** | 法律上真正的爭執焦點（要件是否成立 / 認定是否適法等） |
| **當事人主張** | 納稅義務人（訴願人）主張什麼 |
| **稽徵機關見解** | 國稅局 / 稅捐處（原處分機關）認定什麼 |
| **決定結果** | 如：訴願駁回、撤銷原處分、發回原處分機關另為處分等 |
| **關鍵法條** | 本案援引的主要法規（**若原文有寫才填；原文沒寫就填「原文未載明」，不可自行編造**） |

---

## 三、可直接複製的 Few-shot Prompt（整段貼上）

> 下方【範例一】【範例二】的案例文字為**教學示意（去識別化改寫）**，字號 / 金額皆為示範用途；第三筆之後，你可以換成真正要處理的訴願決定書 / 救濟案例段落。

```
你是稅務救濟文件的「結構化編碼助理」。我會給你一段訴願決定書或租稅救濟案例的文字，請你把它整理成固定欄位。

【規則】
1. 只根據我提供的文字內容抽取，不可加入文字裡沒有的資訊。
2. 「關鍵法條」欄位：只有在文字明確出現法規名稱／條號時才填；文字沒寫就填「原文未載明」，絕對不可自行編造或補上你以為的條號。
3. 若某欄位在文字中找不到，填「未提及」。
4. 用繁體中文、台灣稅務用語。
5. 嚴格照下面範例的欄位與格式輸出，不要多加說明或評論。

======= 範例一 =======
【輸入文字】
訴願人民國113年度綜合所得稅結算申報，列報扶養其胞弟之免稅額，經財政部○○國稅局以其胞弟已成年、當年度有相當工作收入、且未實際同居共財，難認符合受扶養要件，乃否准認列並補徵稅額。訴願人主張其胞弟仍在學、經濟未獨立，實際由其提供生活費用，應准扶養。經審酌，訴願人未能提出確實同居扶養及胞弟無謀生能力之證明，原處分並無違誤。訴願駁回。

【輸出】
案由：綜合所得稅列報扶養胞弟免稅額遭否准，訴願人請求准予認列。
稅目：綜合所得稅
爭點：受扶養親屬（成年胞弟）是否符合「無謀生能力且確係受扶養」之免稅額認列要件。
當事人主張：胞弟仍在學、經濟未獨立，實際由訴願人提供生活費用，應准列報扶養免稅額。
稽徵機關見解：胞弟已成年、當年度有相當工作收入、未實際同居共財，不符受扶養要件，否准認列並補徵。
決定結果：訴願駁回（維持原處分）。
關鍵法條：原文未載明

======= 範例二 =======
【輸入文字】
訴願人為經營餐飲之營業人，經檢舉取具非實際交易對象開立之統一發票申報扣抵進項稅額，原處分機關認定其虛報進項稅額，補徵營業稅並依規定裁處罰鍰。訴願人主張確有進貨事實、貨款亦已支付，縱憑證來源有瑕疵，亦不應連同罰鍰一併處罰。原處分機關則以該等發票非交易對象所開立，屬不得扣抵之憑證，且訴願人未盡查證義務。訴願決定：關於補徵營業稅部分駁回；罰鍰部分因裁量說明不足，撤銷原處分，由原處分機關另為適法之處分。

【輸出】
案由：營業人取具非實際交易對象發票扣抵進項，遭補稅裁罰而提起訴願。
稅目：營業稅
爭點：以非實際交易對象開立之發票申報扣抵進項稅額，可否扣抵及應否裁罰（含有無進貨事實與查證義務）。
當事人主張：確有進貨並已付款，憑證縱有瑕疵亦不應扣抵否准並裁罰。
稽徵機關見解：該發票非交易對象所開立，屬不得扣抵之憑證，訴願人未盡查證義務，虛報進項應補稅裁罰。
決定結果：補徵營業稅部分訴願駁回；罰鍰部分撤銷原處分，發回另為適法處分。
關鍵法條：原文未載明

======= 換你做（第三筆）=======
【輸入文字】
（把要處理的訴願決定書／救濟案例文字貼在這裡）

【輸出】
```

## 四、預期輸出示意

當你在「第三筆」貼進新的案例文字後，AI 會**自動照前兩個範例的格式**吐出七個欄位。例如貼入一段遺產稅漏報存款的案例，它會產出：

> 案由：……（一句話）
> 稅目：遺產稅
> 爭點：……
> 當事人主張：……
> 稽徵機關見解：……
> 決定結果：……
> 關鍵法條：原文未載明（或原文有寫的條號）

---

## 五、批次處理（一次丟多筆）

若要一次處理多筆，把 prompt 最後改成：

```
======= 換你做（多筆）=======
我接下來會一次貼上多段案例，請你為「每一段」各輸出一組上述七個欄位，並在每組前標上「案例編號 1、2、3…」。段落之間我會用「---」分隔。開始：

【案例 1】
（文字）
---
【案例 2】
（文字）
---
【案例 3】
（文字）
```

> 進階（銜接下午試算表）：可要求它「以表格輸出、每列一筆」，就能直接貼進 Excel（詳見 `上午_C_結構化提示範本.md`）。

## 六、常見坑
- **坑 1：範例格式不一致**。兩個範例的欄位名稱、順序、標點要**完全一致**，AI 才會穩定模仿。→ 直接用本檔範例，別手改欄位名。
- **坑 2：AI 幫你「補」法條**。這是幻覺高風險區！→ 規則已寫死「沒寫就填『原文未載明』」，但仍要提醒：**輸出的任何條號都要回原文核對**。
- **坑 3：文字太長被截斷**。→ 調高 Max output tokens，或分批。
- **坑 4：欄位抽取見仁見智（如「爭點」）**。→ 這是內容分析常態；可在規則加一句「爭點請聚焦法律上的認定焦點，非事實細節」以收斂。
- **坑 5：溫度太高**。→ 分類要低溫（0.1~0.3），確保同一段每次分到一樣的類。
